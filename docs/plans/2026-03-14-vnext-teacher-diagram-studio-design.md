# GeoHelper VNext Teacher Diagram Studio Design

Date: 2026-03-14  
Status: Implemented in branch `codex/vnext-teacher-diagram-studio` and validated on 2026-03-15

## Delivery Snapshot

当前分支已落地本设计的第一阶段工作：

1. 首页已切换为教师优先的题图启动页，主入口聚焦“看图/文字快速生成可编辑图形”。
2. 工作区已重排为左侧输入 rail、中间画布、右侧结构化结果 rail 的三栏制图台。
3. 模板库已提升为显式入口，可从首页和输入 rail 打开并直接带入教师场景模板。
4. 证明辅助已作为结果后的下一步动作接入，支持“补辅助线 / 生成讲解思路 / 尝试证明”的 follow-up scaffolding。
5. 视觉系统已从通用浅蓝 SaaS 卡片风格收束为“纸面几何 + 制图台”语义。

## 1. Goals

GeoHelper 下一阶段聚焦成为“中学老师 / 教培老师使用的几何题图制图台”，优先解决以下问题：

1. 把截图题、草图题、文字题快速转成可编辑 GeoGebra 图形。
2. 让老师在生成后继续拖拽、修正、补点、补线，而不是停留在一次性结果。
3. 让“证明辅助”成为结果后的增强能力，而不是打断主流程的独立入口。
4. 用更强的工作台信息架构和视觉语言，明确 GeoHelper 不是普通聊天壳。

成功后的用户心智应当是：

1. 这是一张题图的数字制图台。
2. 我可以把纸面题很快转成课堂可讲的动态图。
3. 生成不是黑盒，我可以看懂步骤、确认不确定项、继续编辑。

## 2. Confirmed Product Direction

Validated decisions from brainstorming:

1. 核心场景优先级：`看图 / 文字快速生成可编辑图形` 高于 `证明辅助`。
2. 主要目标用户：`中学老师 / 教培老师`。
3. 产品气质：`工作台 / 制图台`，而不是聊天工具或教辅门户。
4. 证明辅助的定位：挂在生成结果之后，作为“下一步增强动作”。

## 3. Current Problems

结合当前实现，现阶段主要问题不在基础能力，而在产品表达与核心路径：

1. 首页与工作区仍然以聊天交互为主心智，不能直接突出“题图生成台”的价值。
2. 主工作区把输入、生成结果、继续编辑混在消息流里，结果可执行性不够清晰。
3. 模板入口、图片入口、命令入口分散在 composer 细节里，老师第一次使用成本偏高。
4. 视觉层偏通用工程后台：安全圆角、浅蓝色、卡片化明显，但记忆点不足。
5. 证明辅助虽可存在于消息流里，但还没有和“图已经生成”之后的教师动作形成连续体验。

## 4. Options Considered

### Option A: 通用几何 Copilot

- 优点：场景覆盖面最大，延续现有聊天结构成本最低。
- 缺点：产品辨识度弱，容易被理解为“能画几何图的聊天工具”。

### Option B: 证明助手优先

- 优点：学科深度更强，理论价值明确。
- 缺点：首屏吸引力不如“快速出图”直接，不利于老师高频备课场景。

### Option C: 教师题图制图台

- 优点：最贴近老师真实工作流，价值直观，能把生成、编辑、演示串成闭环。
- 缺点：需要主动重构首页与工作台心智，不能只做表面视觉换皮。

## 5. Recommendation

采用 Option C：GeoHelper VNext 作为“教师题图制图台”。

关键原则：

1. 聊天降级为一种输入方式，不再是产品主角。
2. 画布和“结果可执行性”上升为产品主角。
3. 输入图片、粘贴截图、输入题干必须成为平权入口。
4. 生成后的确认、修正、补图、证明辅助必须进入同一工作链路。

## 6. Target Information Architecture

VNext 信息架构建议拆成 4 个层级：

1. `首页 / 启动页`
   - 负责任务分流与首次启动
   - 不承担复杂编辑
2. `工作台`
   - 完整制图路径：输入 → 生成 → 确认 → 编辑 → 导出
3. `资源库`
   - 模板、最近题图、课堂案例、常用起稿
4. `设置与同步`
   - 保留现有强能力，但退出主创作路径

设计约束：

1. 老师第一次进入时，不需要理解 slash command。
2. 任一入口都应在两步内到达“开始生成图形”。
3. 已有远端备份、运行时、模板等高级能力继续存在，但不抢首屏注意力。

## 7. Homepage Design

首页重构目标：让用户在第一屏就理解“GeoHelper 能把题目变成可编辑图形”。

### 7.1 Core Layout

首页采用左重右轻的非对称结构：

1. 左侧：主输入区
   - 支持拖入题目截图
   - 支持粘贴图片
   - 支持输入文字题干
2. 右侧：教师真实案例区
   - 截图题 → 还原几何关系图
   - 文字题 → 自动起稿
   - 草图题 → 转为课堂演示图

### 7.2 Primary Messaging

首页标题不强调聊天，而强调结果：

- 主标题建议：`把题目变成可编辑几何图`
- 副标题建议：`适合备课、讲题、改题，生成后可继续拖拽与标注`

### 7.3 Primary Actions

一级动作：

1. `开始生成图形`
2. `打开最近图稿`
3. `进入模板库`

交互原则：

1. 不使用“开始聊天”之类泛化文案。
2. 示例应可一键带入，降低首次空白感。
3. 首屏只给 1 个主目标：开始出图。

## 8. Workspace Redesign

工作台从“画布 + 聊天侧栏”升级为“三栏制图台”：

1. 左栏：`输入与任务流`
2. 中栏：`主画布`
3. 右栏：`生成结果与可执行步骤`

### 8.1 Left Rail: Input Panel

左栏不再命名为聊天，而改成输入面板：

1. 顶部模式切换：
   - `看图生成`
   - `文字生成`
   - `继续补图`
2. 中部当前任务输入区：
   - 图片上传 / 粘贴
   - 题干输入
   - 当前任务说明
3. 底部辅助区：
   - 最近输入
   - 推荐模板
   - 当前会话历史

### 8.2 Center: Canvas First

中栏保持绝对主角地位：

1. 画布尺寸优先于消息列表。
2. 支持明显的对象高亮和步骤聚焦。
3. 增加 `演示模式`、`步骤回放`、`导出图稿` 的明确入口。

### 8.3 Right Rail: Structured Result Panel

右栏不再只显示 assistant 气泡，而显示结构化结果：

1. `图形摘要`
   - 系统识别到的点、线、圆、关系
2. `执行步骤`
   - 每一步将创建或修改什么对象
3. `不确定项`
   - 需要老师确认的关系、命名、位置
4. `下一步动作`
   - 补辅助线
   - 生成讲解思路
   - 尝试证明
   - 导出图片 / 课件素材

## 9. Proof Assistance Placement

证明辅助不是首页一级入口，而是生成结果后的增强层。

推荐放置在右栏“下一步动作”中，形成连续工作流：

1. 先得到可编辑图形。
2. 再根据图形关系给出辅助线建议。
3. 再根据题设与目标输出讲解思路或证明草稿。

这样设计的原因：

1. 老师最先需要的是可讲的图，不是抽象证明文本。
2. 有图之后，证明辅助更容易上下文一致。
3. “补辅助线 → 生成讲解思路”天然符合课堂动作顺序。

## 10. Visual System Direction

整体视觉方向定义为：`纸面几何 + 精密制图工具`。

### 10.1 Tone

目标观感：

1. 像老师桌面的制图纸和课件草稿被数字化。
2. 有秩序、有工具感，但不过度冷硬。
3. 适合投屏讲课，也适合长时间备课。

### 10.2 Typography

1. 避免继续依赖通用系统无衬线作为唯一视觉性格。
2. 标题字体应更具几何秩序感。
3. 正文字体应服务长时间阅读与课堂投屏清晰度。

### 10.3 Color

推荐色彩结构：

1. 微暖白或纸面白作为背景。
2. 石墨灰作为主要文本。
3. 墨蓝或绛红作为结构强调色。
4. 高亮色只用于：
   - 当前步骤
   - 待确认项
   - 关键约束关系

应避免继续沿用“浅蓝 SaaS 默认值”作为整体品牌气质。

### 10.4 Spatial Language

1. 减少“处处卡片化”。
2. 左右栏更像工具抽屉，不像消息容器。
3. 首页主输入区像工作纸，案例区像案头参考条。
4. 通过线、留白、分区而不是堆叠卡片建立层级。

### 10.5 Motion

动效应偏工具语义：

1. 短。
2. 干脆。
3. 强调对象吸附、插入、确认。

避免悬浮、发光、漂移式的通用 AI 产品动效。

## 11. Product Surface Changes

VNext 推荐新增或重构以下产品面：

1. 首页主输入启动页。
2. 工作台左栏输入面板。
3. 工作台右栏结构化结果面板。
4. 教师模板库 / 案例库。
5. 演示模式与导出入口。

现有能力的保留方式：

1. 会话历史保留，但弱化为辅助层。
2. 设置与同步保留，后退到工具层。
3. 运行时与模式切换继续存在，但不占首屏主位。

## 12. Phase Plan

### Phase 1

1. 首页重构。
2. 工作台三栏布局。
3. 输入面板职责重组。
4. 基础视觉系统换新。

### Phase 2

1. 结构化结果面板。
2. 不确定项确认流。
3. 教师模板库 / 样例库。

### Phase 3

1. 证明辅助嵌入。
2. 演示模式。
3. 导出与分享能力。
4. 全面视觉 polish。

## 13. Testing Strategy

### 13.1 UX Validation

1. 首次用户是否能在 10 秒内理解产品主价值。
2. 老师是否能在不阅读帮助文档的情况下完成“上传题图 → 生成图形”。
3. 生成结果后的下一步动作是否足够清晰。

### 13.2 Interaction Regression

1. 图片上传、粘贴、拖拽流程。
2. 文本题干输入流程。
3. GeoGebra 生成与继续编辑流程。
4. 历史、设置、同步不回归。

### 13.3 Visual Review

1. 桌面投屏可读性。
2. 长时备课的密度舒适性。
3. 窄屏与短高度布局是否保留核心路径。

## 14. Non-Goals

本阶段不追求：

1. 通用人群泛化聊天助手定位。
2. 独立完整的证明系统首页。
3. 全云端历史或复杂协作体系。
4. 只做皮肤级视觉更新而不改工作流。

## 15. Success Criteria

若 VNext 方向正确，应出现以下结果：

1. 用户会把 GeoHelper 描述为“题图生成和制图工具”，而不是“一个聊天应用”。
2. 首页启动路径更短，首次生成成功率更高。
3. 老师在生成后继续修改图形、补辅助线、导出讲题图稿的频率明显提升。
4. 证明辅助被理解为制图后的自然延伸，而非割裂功能。
